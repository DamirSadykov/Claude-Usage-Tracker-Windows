# Дизайн фичи: интеграция с внешним сервисом задач

Статус: дизайн (код не написан). Целевая ветка: новая, от `master` (master защищён, PR-only).

Все имена конкретных внешних сервисов в этом документе намеренно заменены нейтральными
терминами. Публичный код (resolver + instance) не должен содержать ни названий, ни полей,
ни эндпоинтов конкретного корпоративного сервиса.

## 1. Цель

Показывать в трекере задачи и их изменения (смена статуса и т.п.) из внешнего
корпоративного сервиса, **readonly** — трекер только отображает, ничего не пишет обратно.

Три требования, которые формируют всю архитектуру:

1. **Разделение контрактов.** Публичный (Apache-2.0) проект не должен содержать кода,
   контрактов или секретов конкретного внешнего сервиса.
2. **Чувствительные данные.** В описаниях внешних задач бывают доступы клиента. Они не
   должны покидать периметр — ни в публичный код, ни на диск пользователя.
3. **Readonly** на старте — структурно, а не дисциплинарно.

## 2. Топология

```
[external]          корп-сервис — источник события (смена статуса задачи и т.п.)
   │ push · САНИТИЗАЦИЯ здесь — сырые доступы дальше не идут
   ▼
[external module]   ПРИВАТНЫЙ, вне репо — знает конкретный сервис, нормализует событие в конверт
   │ push · ingest-токен (shared secret) в заголовке, иначе 401
   ▼
[resolver / server] ПУБЛИЧНЫЙ, нейтральный — двойной auth (ingest-токен + подпись устройства),
                     маппинг account→devices, доставка
   │ poll · instance подписывает запрос приватным ключом
   ▼
[instance / desktop] ПУБЛИЧНЫЙ — keypair устройства, readonly-показ, локальный кэш
```

Граница публичное/приватное проходит между `external module` и `resolver`. Всё, что
публично (resolver + instance), — доменно-нейтральное: не знает, что за сервис прислал
событие, и не интерпретирует содержимое.

### Разделение работ

| Компонент | Владелец | Репозиторий |
|---|---|---|
| `external module` — нормализация события в конверт + санитизация + выдача pairing-кода | приватный | вне публичного репо |
| `resolver` — приём конверта, маппинг `account → devices`, доставка по подписи | публичный (я) | этот репо |
| `instance` — keypair, enrollment-клиент, поллинг, кэш, readonly-UI | публичный (я) | этот репо |

## 3. Нейтральный конверт (контракт `external module → resolver`)

Resolver **не читает** `payload` — только маршрутизирует. Поэтому маршрутная информация
лежит отдельно от содержимого.

```
recipient  — account-ключ (корп-email); resolver разворачивает его в список устройств
source     — метка источника ("svc-a") — строка, resolver её не интерпретирует
type       — task_created | task_status_changed | task_moved | task_comment_added
payload    — { title, status, priority?, assignee?, url, updated_at }
             БЕЗ сырого описания задачи — детали пользователь смотрит по url в корп-сервисе
id, ts     — дедупликация и порядок
```

**Инвариант безопасности (требование #2):** отсев доступов — обязанность `external module`
ДО отправки в resolver. По умолчанию сырое описание в `payload` не кладётся вовсе. Resolver
и instance — публичный код, через который текут реальные сообщения; они не должны видеть и
логировать сырые чувствительные данные.

## 4. Идентичность и доставка

### Две сущности, которые нельзя путать

- **account (корп-email)** — *адрес*. Не секрет, может быть известен/угадан. Говорит «кому»,
  не «имеет право».
- **токен/ключ устройства** — *credential*. Забрать сообщения можно только предъявив
  валидную подпись устройства. Запроса «дай задачи для email X» не существует — есть только
  «дай сообщения для моего ключа». Знание чужого email не даёт ничего.

### Один account → много устройств

`personal token` = ключ **устройства**, не пользователя. Два ПК = два ключа под одним
account. Resolver хранит:

```
account (email) → devices: [ { device_id, public_key, enrolled_at, last_seen } ]
mailbox:          [ { id, recipient, source, type, payload, ts, delivered_to: [device_id] } ]
```

Входящее сообщение с `recipient = email` доставляется веером на все устройства аккаунта;
каждое забирает своё при поллинге.

### Keypair (механика доступа)

- Алгоритм — **Ed25519** (`ed25519-dalek`).
- **Приватный ключ** генерится на устройстве, лежит в app data dir, зашифрован **DPAPI**
  (под учётку Windows — не сторонний vault). Не покидает устройство никогда.
- Resolver хранит только **публичный ключ**. Аутентификация поллинга — подпись запроса
  приватным ключом + nonce/timestamp против replay. Утечка базы resolver'а не компрометирует
  ни одно устройство.

### Две границы аутентификации resolver'а

У resolver'а два класса клиентов с разной auth — их нельзя объединять (у отправителя нет
device-ключа, у устройства нет ingest-токена):

```
external module → resolver : ingest-токен (shared secret). Кто вправе КЛАСТЬ. Нет/неверный → 401.
instance → resolver        : подпись устройства (Ed25519). Кто вправе ЗАБИРАТЬ. Нет/неверная → 401.
```

Ingest-токен:
- передаётся в заголовке (`Authorization: Bearer …`) по TLS, не в URL; constant-time
  сравнение; не логируется;
- **по токену на каждый `external module`** (per-source) — точечный отзыв и аудит источника;
  токен привязан к разрешённому `source`, чтобы модуль не мог прикинуться чужим источником;
- resolver принимает **набор** валидных токенов из env/секрет-конфига → ротация без даунтайма.

### Два независимых слоя (чтобы SSO доехал без переделки)

```
слой доступа   : keypair устройства            ← делаем сейчас
слой личности  : доказательство владения account ← сейчас код, позже SSO
```

Enrollment один раз доказывает владение account'ом, дальше доступ идёт по ключу. Смена
способа доказательства (код → SSO) меняет ТОЛЬКО enrollment-шаг; контракт и клиент не
трогаются.

**Enrollment сейчас (код):** instance генерит ключи → показывает pairing-код → пользователь
вводит его в корп-сервисе (где уже залогинен) → корп-сервис вызывает resolver: «этот public
key = account X» → устройство привязано.

## 5. План реализации

Этапы упорядочены так, чтобы каждый давал проверяемый результат и не блокировался приватной
стороной.

- **Этап 0 — Контракт.** Этот документ + JSON-схемы конверта и resolver-store. Зафиксировать
  список `type` событий (следующий артефакт). Приватная и публичная стороны дальше работают
  против него независимо.
- **Этап 1 — Keypair на instance.** Ed25519-генерация, хранение приватного ключа под DPAPI,
  выдача публичного, стабильный `device_id`. Rust в `src-tauri`. Проверяемо юнит-тестом
  подписи/проверки.
- **Этап 2 — Resolver MVP.** Приём конверта от `external module` (аутентификация отправителя —
  см. открытые вопросы), запись в mailbox, endpoint поллинга с проверкой подписи устройства,
  маппинг `account → devices`. Отдельный крейт в репо (кандидат: Rust + axum).
- **Этап 3 — Enrollment.** Endpoint привязки `public key → account` (вызывается корп-сервисом
  по коду), генерация pairing-кода на instance, UI ввода/статуса привязки.
- **Этап 4 — Instance-клиент.** Поллинг resolver'а с подписью запроса, локальный кэш
  `external_tasks.json` (по аналогии с `todos.json`), детект смены статуса сравнением с кэшем.
- **Этап 5 — Readonly-UI.** Представление внешних задач (отдельная панель против вливания в
  доску todos — решается на этом этапе).

Приватная сторона (`external module`: нормализация + санитизация + выдача кода) — вне этого
репо, реализуется параллельно против контракта из Этапа 0.

## 6. Открытые вопросы

- ~~**Список `type` событий**~~ — ЗАКРЫТ: `task_created` | `task_status_changed` | `task_moved` |
  `task_comment_added`; дедуп по `(source, recipient, id)` (fan-out по получателям на стороне источника).
- **Транспорт `resolver → instance`** — поллинг на старте; SSE/WebSocket-push позже, если
  near-real-time понадобится. Контракт от этого не зависит.
- ~~**Аутентификация `external module → resolver`**~~ — РЕШЕНО: ingest-токен (shared secret,
  per-source, ротируемый), в заголовке по TLS, иначе 401. См. §4.
- **Рантайм и хостинг resolver'а** — кандидат Rust + axum, self-hosted (приватный деплой).
  Подтвердить.
- **Retention/дедупликация mailbox** — когда удалять доставленные сообщения.
- **UI-представление** — отдельная панель против вливания в доску todos (Этап 5).
- **SSO** — в разработке у компании; заменит enrollment-код на слое личности, механику ключей
  не трогает.
