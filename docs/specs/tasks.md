# Спека: конвейер задач

> Детальная спека подсистемы задач. Правила жанра — [README.md](README.md).
> Отличие от [task-pipeline.md](../task-pipeline.md): гайд учит «как
> работать с доской», спека фиксирует «как доска себя ведёт».

## §1. Модель и хранение

- Задача — узел направленного ациклического графа. Доски разделены по
  проектам, плюс глобальная доска; задача принадлежит ровно одной.
- У задачи есть UUID и человеко-читаемый номер `#N` (монотонно растёт,
  не переиспользуется). В прозе задача упоминается как `t#N`; голый `#N`
  означает GitHub PR/issue и задачей не считается.
- Хранилище — один файл `todos.json` в data-каталоге приложения
  (`{version: 1, todos: [...]}`). **Единственные писатели — CLI трекера
  и само приложение**; правка файла руками не поддерживается
  (конкурентная запись портит общий файл). Запись атомарна
  (tmp + rename). UI перечитывает доску вживую по изменению файла.

## §2. Поля задачи и их роли

Четыре текстовых поля — четыре роли; один и тот же текст не живёт в двух:

| Поле | Роль |
|---|---|
| `description` | WHAT & WHY; у theme-root — VISION темы (§8) |
| `plan` | только HOW: секции STEPS + ORDER принятого плана; переписывается при перепланировании |
| `handoff` | батон-результат для зависимых задач (§7) |
| `comments` | журнал: решения, смена скоупа, находки; только добавление |

Прочие поля: `status` и `status_history` (лог переходов — питает
стоимость задач, §12), `priority` (high/medium/low/none), `kind` (§6),
`scheduled_for` (дата — попадает в «⏰ due» контекста сессии), `theme`
(§8), `depends_on` и `links` (§4), `project`/`from`, `created_by`.
Пустые поля в файл не пишутся.

## §3. Статусы: две оси

- Колонка (kanban): `backlog | queue | in_progress | review | done` —
  куда задачу положил человек. На графе это заливка узла.
- Pipeline-состояние: `blocked | ready` — **выводится из графа, не
  хранится**. `blocked` = есть незакрытый prerequisite; `ready` = все
  закрыты. На графе — точка в углу узла.
- Оси независимы: задача может быть `in_progress` и `blocked`
  одновременно.

## §4. Рёбра графа

- **dep** (`depends_on`): блокирующее ребро «A ждёт B». Ацикличность и
  принадлежность одной доске проверяются при записи; нарушение —
  отказ. Отсутствующий (удалённый) prerequisite не блокирует.
- **ref** (`links` + inline `t#N` в description/plan): неблокирующая
  ссылка «имей в виду»; может пересекать доски; в вывод blocked/ready
  не входит. Inline `t#N` рисует только ref-ребро — блокирующее ребро
  создаётся исключительно командой `dep add`.
- Критерий выбора: «B должна закончиться до старта A» → dep; «A просто
  хочет видеть B» → ref.

## §5. Инварианты закрытия

- **done-gate**: перевод в `done` отклоняется, пока не закрыт хотя бы
  один **прямой** prerequisite; отказ перечисляет блокирующие задачи.
  Транзитивная честность держится индукцией: честно закрытый прямой
  prerequisite ручается за свой хвост. `--force` — явное исключение
  (устаревший prerequisite, работа сделана вне конвейера); частый
  `--force` означает, что рёбра врут.
- `review` ≠ `done`: prerequisite в review всё ещё держит зависимых.
- `ready` = каждый prerequisite `done`; задача без рёбер — ready.

## §6. Kind: авторитет закрытия

- `manual` (по умолчанию) — человеческий гейт: сессия делает работу, но
  переход `review → done` оставляет пользователю. Красный/жёлтый хвост
  ниже по графу — гейт **работает**, а не сломан.
- `auto` — узел, который сессия может выполнить без присмотра,
  **проверить** и закрыть сама. Ставится только там, где успех
  верифицируем (зелёные тесты, инвариант); ошибочное авто-закрытие
  компаундится вниз по цепочке до ближайшего manual-гейта.
- На графе auto-узел помечен `⚡`.

## §7. Handoff-батон

- `handoff` задачи — что она **произвела**: файлы/пути, интерфейсы,
  решения, грабли, следующий шаг. Пишется для задач, которые от неё
  зависят; батон, не журнал. Пустой допустим.
- Перевод задачи в `in_progress` **автоматически печатает** батоны её
  прямых prerequisites. Читаются только прямые: длинная цепочка контекста
  собирается тем, что батон — авторская проза и сам ссылается на
  `t#N` выше по графу.
- Stop-hook охраняет батон при завершении сессии (§10).

## §8. Темы

- Тема — обычная root-задача, которая `depends_on` всех своих детей;
  отдельной сущности нет, агрегация = dep-рёбра. Из done-gate следует:
  root закрывается последним.
- Root помечается флагом `theme` (`set-theme on` / `add --theme`) —
  флаг нужен потребителям vision, чтобы находить агрегатор
  детерминированно, а не угадывать.
- **Vision темы живёт в description root'а** и читается ВВЕРХ: при
  переводе ребёнка в `in_progress` печатается description ближайшего
  theme-root'а вверх по каждой ветке (внешняя тема поверх внутренней не
  видна); SessionStart-хук повторяет vision для каждой показанной
  `in_progress`-задачи; `todos vision <task>` перечитывает по запросу.
- Handoff root'а — обычный батон вниз тем, кто зависит от самого root'а.
- Хотя рёбра дети→root существуют в данных, на графе Deps тема рисуется
  секцией-аккордеоном, а не терминальным узлом (§13) — стрелки детей в
  root не показываются.
- ⚠ направление (2026-07-21): обсуждается замена темы сущностью
  «спека» — vision живёт в секции спеки, root-агрегатор и флаг theme
  уходят (как ушли фазы); группировка на графе — по спеке. До решения
  темы ведут себя как описано выше.

## §9. Plan mode — ритуал формирования задач

- Хук на входе в plan mode впрыскивает формат плана: абзац VISION
  («должно X, решили Y, потому что Z»), нумерованные STEPS (один шаг =
  одна сессия, план-фразы от первого лица), финальная строка ORDER в
  стрелочной нотации (`Порядок: 1 -> 2; 1 -> 3`), где стрелка — только
  реальный блокер.
- Хук на выходе (план принят) велит записать план в трекер: план из
  нескольких шагов → theme-root (VISION → description) + задача на шаг
  + `dep add` на стрелку + STEPS+ORDER → `set-plan` root'а; план одного
  шага → `set-description` (только на пустое) + `set-plan` самой
  задачи. Один шаг = одна задача, лишние задачи не изобретаются.
- Тот же exit-хук детерминированно гоняет план через `match-plan`
  KB-совместимого CLI, если тот назван в настройках (`matchPlanCli`):
  предупреждения-кейсы впрыскиваются в контекст и просятся в комментарий
  задачи; ноль предупреждений — тишина; сбой матчера никогда не
  блокирует запись.
- Plan mode не обязателен: задача, созданная без него, работает как
  обычно — просто структура не записывается бесплатно.
- ⚠ известный разрыв (2026-07-21): текущий харнес не передаёт текст
  плана в `tool_input` ExitPlanMode (план живёт в файле), поэтому до
  match-plan план не доходит — предупреждения молча выпадают. Чинить —
  чтением плана из `tool_response` или файла.

## §10. Каналы контекста сессии

- **SessionStart-хук**: инструкция по CLI + срез доски проекта — сначала
  ⏰ due/просроченные, затем queue-задачи с закрытыми prerequisites
  (текущие корни графа) и все `in_progress` (с приоритет-гейтом из
  настроек), плюс vision тем для in_progress (§8). Срез ограничен:
  **не более 10 задач**, только заголовки (обрезаются до ~100 символов);
  полная доска — `todos list`.
- **Перевод в `in_progress`**: авто-печать унаследованных батонов (§7)
  и vision (§8).
- **Stop-хук**: блокирует завершение сессии, которая двигала задачу, но
  не оставила свежий содержательный handoff; жалобы конкретны (батон
  пересказывает subject/description; нет следующего шага). Атрибуция
  «сессия двигала задачу» — по транскрипту.

## §11. Триаж

Ночной триаж — встроенный планировщик приложения (не Windows Task
Scheduler): детерминированные три шага — Rust экспортирует срез, агент
читает и пишет разбор, Rust публикует результат.

## §12. Стоимость задач

Сессии атрибутируются задачам по транскриптам и `status_history`;
аналитика показывает стоимость поставленной задачи в токенах/долларах.

## §13. Окно Tasks (UI)

- **Board**: Kanban, 5 колонок, drag-and-drop, живая перезагрузка.
- **Graph, вкладка Deps**: раскладка слева-направо по слоям, полосы по
  проектам; done-предки открытых задач дотягиваются как контекст;
  избыточные (транзитивные) рёбра рисуются бледно. Заливка = статус,
  уголок = blocked (красный) / ready (зелёный auto, жёлтый manual),
  `⚡` = auto, `⊖` складывает эксклюзивное поддерево prerequisites в
  root со счётчиком done/total (складка запоминается локально).
  Drag узел→узел добавляет dep-ребро; правый клик по ребру удаляет.
- **Тема на графе — секция-аккордеон**: развёрнутая тема рисуется
  рамкой вокруг эксклюзивного поддерева root'а; карточка root'а —
  заголовок секции слева, справа члены разложены обычным конвейером;
  рёбра член→root не рисуются (принадлежность показывает рамка), рёбра
  через границу секции рисуются как обычно. `⊖` на карточке root'а
  схлопывает секцию в одну карточку с ▸ done/total. Вложенные темы:
  секцию получает только внешний root, внутренняя тема рисуется узлом.
- **Graph, вкладка Ref**: неблокирующие связи, раскладка по компонентам
  связности.
- Поиск подсвечивает и не фильтрует; выбранный узел подсвечивает свою
  транзитивную цепочку; «Component only» режет вид до компоненты
  выбранной задачи. Хоткеи: Ctrl+F — поиск, Ctrl+P — палитра, поиск по
  номеру задачи поддерживается.
- Уведомления — при переходе задачи в review и done (§4
  [usage-tracker.md](usage-tracker.md)).

## §14. Открытые направления

- t#264 — граф: тема/группа как секция-аккордеон.

  но в коде отсутствует** — либо построить, либо убрать обещание из
  README.
