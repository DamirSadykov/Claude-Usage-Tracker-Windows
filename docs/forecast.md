# Как считается прогноз исчерпания

Документ описывает математику и допущения за расчётом ETA в карточках 5-часовой сессии, 7-дневного лимита и доп. расхода (extra usage). Код — `src-tauri/src/stats/forecast.rs`.

## Что отображается

Каждая карточка показывает четыре величины:

| Поле               | Смысл                                                                                        |
| ------------------ | -------------------------------------------------------------------------------------------- |
| `pct`              | Текущий процент потребления (берётся из последнего снепшота — то же, что показывает API).    |
| `rate_per_hour`    | Скользящее среднее расхода в час за последние 7 дней. Это и есть «темп», по которому считают ETA. |
| `eta_minutes`      | Сколько минут до 100% при текущем `rate_per_hour`.                                           |
| `allowed_per_hour` | %/ч, который можно жечь, чтобы ровно дотянуть до ближайшего ресета.                          |
| `pace`             | `unknown` / `ok` / `warn` — итоговая «оценка». См. ниже.                                     |

Уровень `pace` решается так:

- `unknown` — у нас вообще нет данных для оценки темпа (пустая БД или совсем короткая история).
- `warn` — `eta_minutes < time_to_reset`. До ресета не хватит, надо тормозить.
- `ok` — либо хватает с запасом, либо темп нулевой (idle).

## Источники данных

1. **`usage_snapshots`** — записывается каждые `refresh_interval` секунд (по умолчанию 60с). Содержит `pct`, `reset_at` для каждой шкалы и таймстемп.
2. **`cc_usage`** — события Claude Code, выгребаемые из `~/.claude/projects/<...>/*.jsonl`. Используются как «свидетель idle» (см. ниже).

Lookback по обоим источникам — **7 дней** (`LOOKBACK_HOURS = 168`).

## Алгоритм

### 1. Делим окно на часовые бакеты

Окно `[now − 7 дней, now]` режется на 168 часовых бакетов. Каждый бакет = `[end − 1h, end]`, где `end ∈ {now, now − 1h, ..., now − 167h}`.

### 2. Считаем расход в каждом бакете

Для границ бакета (`start`, `end`) считаем `pct(start)` и `pct(end)`. Функция `pct_at(t)`:

- Если в БД есть снепшот ровно в момент `t` — берём его значение.
- Если `t` между двумя снепшотами с зазором ≤ 3 ч и без падения `pct` — линейная интерполяция.
- Если `t` правее всех снепшотов, но последний снепшот не старше 10 мин — **forward-fill** (значение последнего снепшота). Без этого правая граница самого свежего бакета (`end = now`) всегда отваливалась бы: `Utc::now()` всегда чуть позже последней записи поллинг-цикла.
- Иначе — `None` (бакет не можем оценить по снепшотам).

Если обе границы есть и `pct(end) ≥ pct(start)`, расход в бакете = `pct(end) − pct(start)` (в `%`). Это автоматически `%/ч`, потому что бакет ровно час.

### 3. CC-witness для бакетов без снепшотов

Когда снепшоты бакет не покрывают (приложение было выключено, зазор больше 3 ч, или произошёл ресет внутри), смотрим на `cc_usage`:

- Если в `cc_usage` за весь lookback есть **хоть какая-то** активность (то есть ингест работает) **и** в данном бакете нет ни одного CC-события → бакет = подтверждённый idle, считается как `0 %/ч` и попадает в среднее.
- Если в `cc_usage` пусто вообще (CC не используется или ingest сломан) → fallback: бакет дропается. Иначе мы бы выдумывали idle.

Поиск активности — `partition_point` по сортированному массиву таймстемпов, `O(log n)` на бакет.

### 4. Среднее по покрытым бакетам

```
rate_per_hour = sum(bucket_rates) / count(covered_buckets)
coverage_hours = count(covered_buckets)
```

Если `coverage_hours == 0` — возвращаем `unknown`.

### 5. ETA и allowed pace

```
eta_minutes = (100 − current_pct) / (rate_per_hour / 60)   при rate_per_hour > 0
allowed_per_hour = (100 − current_pct) / time_to_reset_min · 60   при reset_at заданном
```

## Обработка ресетов

`pct` шкалы периодически падает в 0 (5-часовая сессия — каждые 5ч, недельная — раз в 7 дней). Ресет ловится по **падению `pct` между bracket-снепшотами**, а не по сравнению строк `reset_at`. Причина: API отдаёт `resets_at` с микросекундной точностью, которая дрейфует между запросами. Старая версия сравнивала строки и сбрасывала почти каждую пару, постоянно показывая «собираю данные…».

Если `pct_at` видит, что между его bracket-снепшотами `pct` упал → возвращает `None` для этого момента. Соответствующий бакет не учитывается (или, если в этот час нет CC-активности, учитывается как 0 через CC-witness).

## Почему скользящее среднее, а не короткое окно

Раньше темп считался по короткому окну (60 минут):

```
rate = (pct_latest − pct_60min_ago) / 60 минут
```

Это даёт «темп текущей сессии», который для 5-часовой шкалы ещё разумен, но для 7-дневной — нет: один активный час 20%/ч экстраполировался на «исчерпается за 5 часов», хотя реально пользователь работает 4 часа в день и у него ещё дней пять впереди.

Скользящее среднее за 7 дней амортизирует расход: 20%/ч в одном часу при остальных 167 нулевых даёт `20/168 ≈ 0.12 %/ч` в перспективе. Это и есть честный прогноз «если буду использовать как обычно».

Медиана пробовалась первой и показала противоположный перекос: для типичного пользователя медиана 168 часов = 0 (idle часов больше половины), ETA всегда отсутствует, прогноз вырождается в «хватит до сброса» навсегда. Среднее устойчивее.

## Почему `MIN_RATE` в forecast.rs нет

`alerts::engine` фильтрует темпы ниже `0.05 %/мин ≈ 3 %/ч` как шум — это корректно для короткого окна предупреждения «5-часовая сессия скоро закончится». Для среднего за неделю типичный темп ~0.1–1 %/ч; фильтр в 3 %/ч давил бы любую реальную оценку. Решение «показывать ли warn» делает проверка `eta < ttr`, а не отдельный порог.

## Константы и их обоснование

| Константа              | Значение | Откуда                                                                                            |
| ---------------------- | -------- | ------------------------------------------------------------------------------------------------- |
| `LOOKBACK_HOURS`       | 168 ч    | Покрывает самую медленную шкалу (7д) и сглаживает суточные/недельные idle-паттерны.               |
| `MAX_INTERP_GAP_MIN`   | 180 мин  | Покрывает обед/митинг, не выдумывая данные за длинные простои.                                    |
| `FORWARD_FILL_GAP_MIN` | 10 мин   | Запас на дрейф `Utc::now()` относительно последнего снепшота при `refresh_interval ≥ 10s`.        |

## Allowed pace и pace tick на полоске

`allowed_per_hour` считается независимо от истории — только из текущего `pct` и `reset_at`. Это «идеальный темп, чтобы дотянуть». Для недельной полоски ещё рисуется `pace-tick` — где «должен» быть текущий процент при равномерном расходе. Считается во фронте (`UsagePanel.vue:idealPace`).

## Ограничения

- **Web/API без Claude Code не виден**. `cc_usage` пишется только из CLI-транскриптов. Пользователь, который жжёт лимит через [claude.ai](https://claude.ai) при выключенном CC, будет получать заниженный прогноз. Это лучше прежнего перекоса в обратную сторону, но об этом стоит помнить.
- **Свежеустановленный билд** — нужна история. До накопления хотя бы одного часа покрытия (с учётом forward-fill это первый же успешный поллинг + ~1 час назад) карточка покажет `unknown`.
- **Длинный простой**. Если приложение было выключено более 3 часов, а в этот период не было CC-активности, бакеты этих часов засчитываются как 0. Если CC-активность была — бакеты дропаются. Если в `cc_usage` вообще пусто — снова дроп.
- **5-часовая сессия**. Та же логика, но 168 бакетов недельного среднего сглаживают короткие пики. Это сознательное упрощение: пользователь хотел одинаковый алгоритм для обеих шкал. Алёрт-движок (`alerts::engine`) использует короткое окно отдельно, для нотификации «сессия скоро кончится».
