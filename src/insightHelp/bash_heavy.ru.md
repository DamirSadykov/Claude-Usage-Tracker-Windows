## Что меряет

Bash составил ≥40% от всех вызовов инструментов за период (при ≥50 вызовах суммарно). Это observation, не recommendation — само по себе не плохо, но стоит понимать, что происходит.

## Почему это важно

Bash — самый «шумный» инструмент по влиянию на контекст. Каждый запуск возвращает свой stdout/stderr в историю целиком, и длинные команды (`cargo build`, тесты, `git log -p`) могут вылить 5–20K токенов в префикс за один вызов. На длинной сессии накапливается быстро.

Распространённые сценарии Bash-доминирования:

- **CI/тесты в цикле** — нормально, если ты гоняешь сборку. Просто следи за общим контекстом.
- **Дебаг через `cat`/`grep`** — антипаттерн. Для этого есть Read и Grep, они дешевле и не загрязняют историю длинным stdout.
- **`find` вместо Glob** — Glob возвращает только список путей, find через Bash тащит свой шумный вывод.

## Что делать

- Замени `cat file.ts` → Read, `grep -r pattern` → Grep, `find ... -name` → Glob.
- Длинные команды (`npm test`, `cargo build`) запускай с пайпом в `tail` или `2>&1 | head -n 50`, если выход не нужен целиком.
- Долгие команды — в `run_in_background: true`. Тогда вывод приходит уведомлением, а не дописывается в префикс.
- Если правда нужна полная сессия debug в Bash — оцени, не пора ли её вынести в отдельный чат под одну фазу.

## Что обычно видно при доминировании Bash

В длинных сессиях с большой долей Bash обычно набегают сотни запусков с длинным stdout (тесты, `cargo build`, `git log -p`, рекурсивный `grep`). Каждый такой вывод оседает в основном префиксе до конца сессии. Если хотя бы половина Bash-вызовов заменяется на Grep/Read/Glob, средний контекст по сессии падает заметно — иногда на десятки процентов.
