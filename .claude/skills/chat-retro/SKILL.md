# chat-retro — ретроспектива по чату

Разбор одного transcript'а Claude Code по его GUID. На выходе — короткий отчёт в три блока: **что хорошо**, **что плохо**, **что менять**.

Не аналитика, не дашборд — именно ретро: что эта конкретная сессия сделала по уму, а что зря. Короткие предложения, конкретные примеры. Без оценок самого себя или пользователя — просто наблюдения по цифрам.

## Когда запускать

Только когда пользователь прямо просит — фраза вроде:
- `/chat-retro <guid>`
- «разбери чат `<guid>`»
- «ретроспектива по `<guid>`»
- «прогони ретро по сессии `<guid>`»

Если пользователь просто закидывает GUID без контекста — уточни одной фразой, нужна ли ретроспектива или просто найти файл.

## Где лежит чат

Каждый чат — это файл `<guid>.jsonl` где-то под `~/.claude/projects/<encoded-cwd>/`. Найди через Glob:

```
pattern: **/<guid>.jsonl
path: ~/.claude/projects/
```

Если ничего не нашлось — сообщи пользователю и не выдумывай данные.

## Что считать

JSONL — это построчный JSON. Каждая строка — один объект. Важные типы:
- `type: "assistant"` с `message.usage` — токены и модель
- `type: "user"` — сообщения пользователя (и tool_result'ы внутри `content[]`)

**Важно про дубли:** один `message.id` может встречаться в нескольких строках JSONL (по строке на каждый content-block). При суммировании токенов **дедуплицируй по `message.id`** — иначе насчитаешь в 2-3 раза больше.

Что собрать для отчёта:

| Метрика | Откуда |
|---|---|
| Период (start → end) | первый и последний `timestamp` |
| Сколько ходов ассистента | distinct `message.id` |
| Сколько user-промптов | строки `type: "user"` с непустым `content[].text` (выкинуть `<system-reminder>` и tool_result'ы) |
| Input / Output / Cache create / Cache read | сумма по дедуплицированным сообщениям |
| Стоимость | по формуле модели (Opus: in $15/M, out $75/M, cache write $18.75/M, cache read $1.50/M) |
| Tool-use counts | пройтись по `message.content[]`, найти блоки `type: "tool_use"`, сгруппировать по `name` |
| Средний контекст на ход | `cache_read / messages` (приблизительно) |
| Пиковый контекст | максимальный `cache_read` среди ходов |
| Топ-5 дорогих ходов | сортировка по стоимости каждого хода |
| Самые длинные ответы | сортировка по `output_tokens` |
| Авто-компакции | поиск в user-сообщениях строк `This session is being continued`, `Previous Conversation Compacted`, `Conversation summarized` — посчитать сколько раз |
| Топ файлов по числу Read | пройтись по tool_use блокам `Read`, вытащить `input.file_path`, сгруппировать по basename, top-5 |
| Прерывания | посчитать вхождения `[Request interrupted by user]` в user-сообщениях |
| Холодные старты в середине | ходы с `cache_read = 0` И `cache_create > 10К` (порог калиброван: реальные компакции дают median 24К, p25 ~20К, ниже 10К — шум), исключая первый ход сессии |

Скрипт можно собрать сразу — PowerShell на Windows, Bash+jq на Linux/Mac. Для дедупа по `message.id` достаточно `HashSet`.

## Как интерпретировать (для блоков отчёта)

Не выдавать всё подряд — выбирать 2-4 пункта в каждый блок, по тому что реально выпирает в этой сессии. Если нечего сказать в блок — оставить «—», не натягивать.

### Что хорошо — что искать

- **Контекст не раздувался** — большинство ходов в `cache_read < 100К` → можно было долго работать без рестарта
- **Кэш стабильный** — `cache_create / (cache_create + cache_read) < 10%` → контекст не инвалидировался
- **Один фокус** — все user-промпты на одну тему, нет резких переключений
- **Дешёвая работа** — медианная стоимость хода < $0.10
- **Edit-heavy, не Read-heavy** — много `Edit`, мало повторных `Read` одних и тех же файлов
- **Сабагенты сократили основной поток** — если использовались, основной кэш не разрастался
- **Короткие чёткие первые промпты** — < 200 символов, конкретный артефакт или вопрос
- **Сессия завершилась в адекватном месте** — последний ход не висит на половине задачи

### Что плохо — что искать

- **Раздутый контекст** — > 50% ходов в `cache_read > 150К` → каждый ход уже дорогой
- **Частые перезаписи кэша** — `cache_create` > 15% от всего кэша → правил файлы / системку в середине
- **Длинная сессия** — > 8 часов или > 300 ходов → хвостовая цена на каждом ходу
- **Авто-компакции** — ≥ 2 раз → сессия пережила автосжатие. Каждая компакция = $3-5 чистых потерь на холодный старт + просадка качества хвоста. Compact — это «не нажал тормоз вовремя»
- **Холодные старты в середине** — ходы с `cache_read = 0` и большим `cache_create` ВНЕ первого хода сессии → или compact, или смена модели. Часто видны как топ-ходы по стоимости с коротким output
- **Многословные ответы вместо действий** — топ-ход по стоимости с `output > 5000` и без tool_use → план который никто не просил
- **Повторные Read одних файлов** — называть конкретные файлы и числа: «`App.vue` × 39», «`SettingsPanel.vue` × 27». Полный Read дорогого файла при `cache_read > 150К` — самый прямой драйвер раздувания
- **Смена моделей в середине** — присутствие 2+ Opus-версий → холодный кэш при каждом переключении; считать число переходов, а не только факт смены
- **Много прерываний** — общее число `[Request interrupted]` важно: 1-2 — норма, 10+ — паттерн «отправил, передумал, переформулировал», обычно совмещается с переотправками
- **Пользователь переформулировал** — один и тот же запрос дважды в коротком интервале → не понял с первого раза
- **`[Request interrupted]` в начале** — несколько прерванных стартов, прежде чем пошла работа

### Что менять — рекомендации

Каждая рекомендация должна указывать на конкретное «плохо» сверху и говорить, что делать в следующий раз. Не общие принципы.

Шаблоны:
- «Когда контекст переваливает за 150К — рестарт чата, продолжать дороже»
- «`<имя файла>` читался N раз — в следующий раз использовать `offset/limit` или Grep»
- «Сессия росла 12 часов — после блока задач X стоило начать новую»
- «Было N авто-компакций (~$N всего) — после первой компакции `/clear` + короткий карри-овер был бы дешевле»
- «Топ-3 дорогих хода — все посткомпактные холодные старты на коротких Bash — ровно эти $X можно было не платить»
- «На ход <id> ушло $3 на план — план в одно предложение + сразу Edit был бы дешевле»
- «5 переходов opus-4-6 ↔ opus-4-7 = 5 холодных стартов на $N — закреплять одну модель на сессию»
- «N прерываний за сессию — собирать промпт целиком в редакторе перед отправкой, не дописывать после старта»
- «Промпт «<цитата>» пришлось переформулировать — в следующий раз сразу добавлять <чего не хватило>»

Если в сессии ничего критичного не было — пиши «менять нечего, нормальный рабочий чат», не натягивай.

## Формат вывода

Печатать прямо в чат. Не сохранять файл по умолчанию — спросить в конце, нужно ли. Размер отчёта — экран, не больше.

```markdown
# Ретро: <guid>

**Где:** <путь к JSONL>
**Когда:** <start> → <end> (<длительность>)
**Проект:** <basename папки>
**Размер:** N ходов ассистента, M user-промптов, ~$X стоимость

## Что хорошо
- <пункт 1>
- <пункт 2>
- ...

## Что плохо
- <пункт 1 с конкретной цифрой>
- <пункт 2>
- ...

## Что менять
- <рекомендация 1 — указывает на пункт из «плохо»>
- <рекомендация 2>
- ...
```

После вывода — одна строка: «Сохранить отчёт? (да → укажу куда: KB / scratch / рядом с проектом)».

## Тон

- Короткие предложения. Где можно — одно вместо двух.
- Никаких «возможно следовало бы рассмотреть» — пиши прямо.
- Цифры с примерами: «$3.20 за один ход на план» лучше чем «один из ходов был дорогим».
- Если повторяющаяся проблема — назвать одним именем («раздутый контекст», «холодные старты»), не пересказывать механизм каждый раз.
- Не лестно и не самобичевательно — нейтрально, как code review.

## Если чат маленький

Сессия из < 20 ходов — отчёт совсем компактный. По 1-2 пункта в каждый блок, или «—» если нечего сказать. Не растягивай ради структуры.
